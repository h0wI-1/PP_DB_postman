# Отчёт по лабораторной работе №4
## Тестирование базы данных через Postman

---

## 1. Цель работы

Научиться проектировать простую реляционную базу данных, подключать её к REST
API и выполнять тестирование запросов к базе с помощью Postman. В ходе работы
необходимо показать выполнение единичных запросов, запросов с параметрами,
стандартных операций CRUD, собрать всё это в коллекцию, добавить раздел по
безопасности и авторизации по токену, а также написать автотесты.

В качестве предметной области выбран интернет-магазин техники. База данных
включает четыре таблицы: клиенты, товары, статусы заказов и сами заказы.

---

## 2. Используемые инструменты

- **PostgreSQL 14** — реляционная СУБД, в которой хранится база данных.
- **PostgREST** — надстройка над PostgreSQL, которая автоматически публикует
  таблицы и функции базы данных в виде REST API. Для проекта выбрана именно
  эта связка, потому что она даёт минимальный по объёму код и при этом
  позволяет показать все необходимые типы запросов.
- **Postman** — клиент для HTTP-запросов, через который выполняется
  тестирование.
- **DBeaver** — графическая среда для работы с базой данных, использовалась
  для проверки результатов SQL-запросов и построения диаграммы.
- Расширение **pgjwt** для PostgreSQL — используется для подписи JWT-токенов
  в функции авторизации.

## 3. Создание базы данных

База данных `shop_db` состоит из четырёх таблиц:

- `clients` — информация о клиентах магазина;
- `items` — товары, которые продаёт магазин;
- `order_status` — справочник статусов выполнения заказа;
- `orders` — заказы клиентов.

Таблица `orders` связана с таблицами `clients`, `items` и `order_status`
внешними ключами. Первичные ключи во всех таблицах — целочисленные
последовательности (`SERIAL`). Для полей, по которым выполняется фильтрация
(`items.price`, `clients.name`), добавлены индексы.

SQL-скрипт создания таблиц приведён ниже.

```sql
CREATE TABLE clients (
    id             SERIAL PRIMARY KEY,
    name           VARCHAR(100) NOT NULL,
    email          VARCHAR(120) NOT NULL UNIQUE,
    phone          VARCHAR(20),
    city           VARCHAR(80),
    registered_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE items (
    id        SERIAL PRIMARY KEY,
    name      VARCHAR(120)   NOT NULL,
    category  VARCHAR(60)    NOT NULL,
    price     NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
    stock     INTEGER        NOT NULL DEFAULT 0
);

CREATE TABLE order_status (
    id     SERIAL PRIMARY KEY,
    code   VARCHAR(20) NOT NULL UNIQUE,
    title  VARCHAR(60) NOT NULL
);

CREATE TABLE orders (
    id          SERIAL PRIMARY KEY,
    client_id   INTEGER NOT NULL REFERENCES clients(id),
    item_id     INTEGER NOT NULL REFERENCES items(id),
    status_id   INTEGER NOT NULL REFERENCES order_status(id),
    quantity    INTEGER NOT NULL DEFAULT 1,
    total       NUMERIC(12, 2) NOT NULL,
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

После создания таблицы были заполнены тестовыми данными: 6 клиентов,
8 товаров, 5 статусов и 6 заказов. Данные подобраны так, чтобы запросы из
следующего раздела сразу возвращали осмысленный результат.

## 4. Подключение базы данных к API

Для того чтобы обратиться к базе данных через HTTP, использована утилита
PostgREST. Её настройки хранятся в файле `postgrest.conf`, который содержит
параметры подключения:

```
db-uri      = "postgres://postgres:postgres@localhost:5432/shop_db"
db-schema   = "public"
db-anon-role = "web_anon"
jwt-secret  = "super_secret_key_change_me_please_32chars"
server-port = 3000
```

- `db-uri` задаёт строку подключения к базе;
- `db-schema` указывает схему, таблицы которой будут опубликованы;
- `db-anon-role` — роль, под которой выполняются запросы без токена;
- `jwt-secret` — секрет для проверки JWT-токенов;
- `server-port` — порт, на котором PostgREST будет слушать запросы.

После запуска команды `postgrest postgrest.conf` в терминале появляется
сообщение «Listening on port 3000», что говорит о готовности сервера. Проверка
доступности API сделана простым открытием URL `http://localhost:3000/clients`
в браузере: в ответ приходит JSON со списком клиентов.

### 5.2. GET-запросы с параметрами

В PostgREST параметры фильтрации передаются в URL после знака `?`. Синтаксис
параметра выглядит как `поле=оператор.значение`. Использованы следующие
примеры:

- `GET /clients?id=eq.1` — вернёт одного клиента с `id = 1`.
- `GET /items?price=gt.40000` — вернёт только те товары, у которых цена
  больше 40 000.
- `GET /clients?name=like.A*` — вернёт клиентов, чьё имя начинается на «A»
  (оператор `like` с шаблоном, где `*` соответствует `%` в SQL).
- `GET /clients?select=name` — ответ содержит только поле `name`, остальные
  столбцы в выдаче скрыты.

Все перечисленные запросы выполнены в Postman, результаты соответствуют
ожидаемым.

### 5.3. Стандартные запросы POST, PATCH, DELETE

В задании встречается запрос `PULL`. В терминологии HTTP такого метода нет;
ближайшие по смыслу — `PUT` и `PATCH`. PostgREST поддерживает оба, но для
частичного обновления записи используется именно `PATCH`, поэтому в работе
применяется он.

**POST — добавление клиента.** Метод `POST`, адрес `/clients`, тело запроса в
формате JSON:

```json
{
    "name":  "Test User",
    "email": "test.user@example.com",
    "phone": "+7-900-000-00-00",
    "city":  "Тестоград"
}
```

В заголовках указано `Prefer: return=representation`, чтобы PostgREST вернул
созданную запись целиком вместе со сгенерированным `id`. Сервер отвечает
статусом `201 Created`.

**PATCH — изменение телефона.** Метод `PATCH`, адрес
`/clients?id=eq.{new_id}`, тело запроса:

```json
{ "phone": "+7-900-999-99-99", "city": "Москва" }
```

После выполнения `PATCH` повторный `GET` показывает, что поле действительно
обновилось.

**DELETE — удаление записи.** Метод `DELETE`, адрес `/clients?id=eq.{new_id}`.
Для проверки удаления выполняется повторный `GET` с тем же параметром:
в ответ приходит пустой массив `[]`, что подтверждает отсутствие записи.

## 6. Создание коллекции запросов и её запуск

Все отлаженные запросы объединены в коллекцию Postman под названием
«Shop API». Для удобства коллекция разбита на папки:

1. **01. Авторизация** — запрос получения токена.
2. **02. GET-запросы без параметров** — базовые запросы.
3. **03. GET-запросы с параметрами** — запросы с фильтрацией.
4. **04. CRUD** — последовательное добавление, изменение и удаление
   клиента с проверками.

Чтобы запросы не зависели от жёстко прописанных значений, создан environment
«Shop API Local» с переменными `base_url`, `token`, `new_client_id`,
`login_email`, `login_password`. В самих запросах вместо конкретных значений
используются подстановки `{{base_url}}`, `{{token}}` и т. д.

Для запуска коллекции использован встроенный в Postman **Collection Runner**:
`Runner` → выбрать коллекцию «Shop API» → `Run`. Запросы выполняются по
порядку, что важно для CRUD-блока: сначала `POST` кладёт `id` нового клиента
в переменную, затем `PATCH`, `DELETE` и проверочные `GET` используют эту
переменную.

## 7. Безопасность БД и авторизация по токену

По умолчанию PostgREST выполняет любой запрос под анонимной ролью
(`db-anon-role = web_anon`). В проекте эта роль специально ограничена: ей
выдан только `SELECT` на все четыре таблицы. Добавление, изменение и удаление
данных доступны только роли `web_user`. Чтобы запросить работу под этой
ролью, нужен валидный JWT-токен.

Схема авторизации следующая:

1. В базе создана таблица `users` с логинами и хэшами паролей (хэширование
   через `crypt()` из расширения `pgcrypto`).
2. Создана хранимая функция `public.login(email, password)`. Она проверяет
   пароль и, если он верен, возвращает JWT, подписанный секретом
   `jwt-secret`. В payload токена записана роль `web_user` и срок действия.
3. Клиент (Postman) отправляет `POST /rpc/login` с email и паролем, в ответ
   получает токен.
4. Во всех последующих запросах токен передаётся в заголовке
   `Authorization: Bearer <token>`. PostgREST проверяет подпись, извлекает
   роль из payload и выполняет запрос уже от имени `web_user`.

В Postman этот процесс настроен так:
- На уровне коллекции выбрана авторизация **Bearer Token** со значением
  `{{token}}`.
- В тестах запроса `POST /rpc/login` есть строка
  `pm.environment.set('token', body.token);`, которая автоматически
  подставляет полученный токен во все последующие запросы.

Таким образом, «секретный ключ» (JWT) скрыт в переменной окружения
Postman и не показывается в теле других запросов.

## 8. Автотесты в Postman

Для каждого запроса коллекции во вкладке **Tests** написан набор проверок на
языке JavaScript. Проверки сделаны так, чтобы при изменении данных в базе
тесты не падали зря, но при этом реально контролировали работу API.

Типовой набор проверок для GET-запроса:

```javascript
pm.test('Status 200', function () {
    pm.response.to.have.status(200);
});
pm.test('Response time < 1000ms', function () {
    pm.expect(pm.response.responseTime).to.be.below(1000);
});
const body = pm.response.json();
pm.test('Ответ — непустой массив', function () {
    pm.expect(body).to.be.an('array').with.length.above(0);
});
pm.test('У клиента есть ключевые поля', function () {
    pm.expect(body[0]).to.have.all.keys(
        'id', 'name', 'email', 'phone', 'city', 'registered_at'
    );
});
```

Для POST-запроса дополнительно проверяется код 201 и наличие `id` в ответе,
после чего `id` сохраняется в переменную окружения:

```javascript
pm.test('Status 201', function () {
    pm.response.to.have.status(201);
});
pm.environment.set('new_client_id', body[0].id);
```

Для PATCH-запроса проверяется, что возвращённая запись действительно
содержит новое значение поля. Для DELETE-запроса проверяется код ответа, а
следом выполняется проверочный GET, который должен вернуть пустой массив.

При запуске коллекции через Collection Runner все проверки выполняются
подряд, и в итоговой сводке видно общее количество пройденных тестов.

В ходе выполнения работы была спроектирована и создана учебная база данных

удаления записей, а также способ проверки результата каждой операции через
повторный GET.

Отдельно реализован раздел по безопасности: ограничены права анонимной
роли, создана функция логина, которая возвращает JWT, и настроена
авторизация в Postman через заголовок `Authorization: Bearer`. Все запросы
собраны в единую коллекцию и покрыты автотестами, что позволяет одним
нажатием кнопки убедиться в работоспособности всей цепочки запросов.

В результате получен рабочий стенд, который можно использовать как основу
для дальнейшего изучения REST API и автоматизированного тестирования.
