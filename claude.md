# CLAUDE.md

## Проект
Учебный проект по Postman / БД / REST API / автотестам.

## Цель проекта
Реализовать и оформить учебную работу по тестированию базы данных через Postman с последующим написанием отчёта в студенческом стиле.

## Основная тема
Интернет-магазин техники.

## Базовая предметная область
Использовать 4 таблицы:
- `clients`
- `items`
- `order_status`
- `orders`

## Что требуется сделать
1. Создать БД.
2. Подключить БД к API.
3. Проверить БД единичными запросами в Postman.
4. Показать GET-запросы с параметрами после `?`.
5. Показать стандартные запросы:
   - GET
   - POST
   - PATCH/PUT
   - DELETE
6. Создать collection в Postman и запустить её.
7. Реализовать раздел по безопасности БД:
   - получение токена
   - авторизация через Bearer Token
8. Сделать автотесты в Postman.
9. Подготовить отчёт.
10. Продумать и оформить скриншоты.

## Что видно из материалов преподавателя
- Используется PostgreSQL.
- Для подключения БД к API в образце показан PostgREST.
- В образце фигурирует `postgres.conf` с параметрами:
  - `db-uri`
  - `db-schema = "public"`
  - `db-anon-role`
  - `jwt-secret`
  - `server-port = 3000`
- Показаны GET-запросы:
  - `/clients`
  - `/items`
- Показаны параметризованные запросы:
  - `clients?id=eq.1`
  - `items?price=gt.40000`
  - `clients?name=like(all).{A*}`
  - `clients?select=name`
- DELETE проверяется повторным GET.
- В разделе безопасности показан hidden API key / secret key и Bearer Token в Postman.

## Рабочее предпочтение
По умолчанию придерживаться архитектуры:
**PostgreSQL + PostgREST + Postman**

Если предлагается Supabase, это нужно отдельно обосновать. При этом итоговая логика, структура отчёта и демонстрация должны оставаться максимально близкими к образцу преподавателя.

## Важное уточнение
Если в задании написано `PULL`, трактовать это как `PATCH` или `PUT`. Для PostgREST предпочтительно использовать `PATCH`, но это нужно аккуратно пояснить в тексте работы.

## Требования к ответам
- Давать не общие советы, а рабочие материалы.
- Сразу писать готовые SQL-скрипты, HTTP-запросы и тесты.
- Все объяснения — на русском языке.
- Текст отчёта — как у студента, без упоминания ИИ.
- Если есть неоднозначность, выбирать самый практичный вариант и коротко объяснять причину.

## Требования к отчёту
Структура:
1. Цель работы
2. Используемые инструменты
3. Создание базы данных
4. Подключение базы данных к API
5. Тестирование базы данных через Postman
6. Создание коллекции запросов и её запуск
7. Безопасность БД и авторизация по токену
8. Автотесты в Postman
9. Вывод

Для каждого раздела нужно:
- готовый текст
- указание нужного скриншота
- подпись вида `Рисунок N – ...`

## Требования к скриншотам
- Только белый фон
- Без тёмной темы
- Чистый и читаемый интерфейс
- Единый стиль подписей
- Если скриншоты нельзя реально сгенерировать, обязательно давать точный shot list:
  - что открыть
  - что должно быть видно
  - что выделить
  - какая будет подпись

## Требования к автотестам Postman
Минимально нужно покрыть:
- status code
- response time
- непустой ответ
- проверка ключевых полей JSON
- POST -> запись добавилась
- PATCH/PUT -> данные изменились
- DELETE -> запись больше не возвращается
- последовательный прогон collection
